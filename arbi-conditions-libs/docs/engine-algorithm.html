Strategy Engine — как работают три функции и условия

Блок-схемы и разбор алгоритма: processStep оценивает шаг по окну, prepareSteps готовит минимальное окно, processAllStepsAndRecordResults прогоняет всю историю. Все три опираются на общий реестр условий CONDITIONS.

pure TypeScript framework-agnostic registry: 6 conditions aggregation: AND
Общая картина

Данные против правил, по окну шагов

Движок берёт окно рыночных шагов (steps, последний = текущий) и стратегию (strategy — пороги/флаги). Каждое условие сравнивает данные с правилами и возвращает { passed, actual, required }. Сторона (buy/sell) проходит, если прошли все её условия.

Роли трёх функций

processStep — ядро. Оценивает один текущий шаг по окну.

prepareSteps — режет всю историю до минимального окна и отдаёт готовый вход для processStep.

processAllStepsAndRecordResults — прогоняет историю шаг за шагом растущим окном, собирает записи.

Строительный блок

ConditionDef — самоописываемое условие

Каждое условие — объект с двумя методами: window() говорит, сколько истории нужно (для prepareSteps), а evaluate() — как считаться (для processStep). Всё лежит в массиве CONDITIONS; добавить условие = дописать один объект.

Контракт

interface ConditionDef {
  id: ConditionId;
  // сколько истории надо
  window(strategy, side): WindowRequirement;
  // как оценить текущий шаг
  evaluate(ctx, strategy, side): ConditionOutcome;
}

type WindowRequirement = {
  steps?: number;           // N последних шагов
  durationMs?: number;      // окно по времени
  toLastTransaction?: boolean;
};

type ConditionOutcome = {
  passed: boolean;
  actual?: number | string; // что наблюдали
  required?: number | string;// с чем сравнили
};

Контекст оценки

interface EvalContext {
  window: MarketStep[];  // шаги до текущего вкл.
  current: MarketStep;   // последний в window
  position: PositionState | null;
}

// пример: spread_ok
{
  id: 'spread_ok',
  window: () => ({}),        // только текущий шаг
  evaluate: (ctx, strategy, side) => {
    const spread = pctDiff(
      ctx.current.quotes.buyQuote,
      ctx.current.quotes.sellQuote);
    const max = strategy[side].maxBuySellSpreadPercent;
    return { passed: spread <= max,
             actual: spread, required: max };
  },
}

Функция 1 · ядро

processStep — оценка текущего шага

Определяет окно (по currentIndex), берёт текущий шаг, строит контекст и прогоняет все условия для каждой стороны, агрегируя через AND.

↻ evaluateSide(side) — повторяется для buy и для sell

↻ по каждому условию реестра

Ядро агрегации — evaluateSide

const evaluateSide = (side) => {
  const outcomes = {};
  let passed = true;
  for (const def of CONDITIONS) {
    const outcome = def.evaluate(ctx, strategy, side);
    outcomes[def.id] = outcome;
    if (!outcome.passed) passed = false;  // AND
  }
  return { outcomes, passed };
};

Функция 2 · подготовка

prepareSteps — минимальное окно из истории

Спрашивает у каждого условия его window(), берёт максимум требований и режет историю до самого раннего нужного индекса. Возвращает готовый ProcessStepParams.

start
prepareSteps({ steps, strategy, currentIndex?, position? })
decision
steps.length === 0 ?
да → return { steps, strategy, position } (пусто пройдёт насквозь)
step
ci = clamp(currentIndex ?? last, 0, last)
history = steps.slice(0, ci + 1) · отбрасываем будущее · current = history[last]
для def × side ∈ (buy, sell)
↻ агрегируем требования по всему реестру
req = def.window(strategy, side)
neededSteps = max(neededSteps, req.steps) · maxDurationMs = max(…, req.durationMs) · toLastTransaction ||= req.toLastTransaction
keepFrom — самый ранний индекс
rule · count
keepFrom = max(0, last − (neededSteps − 1))
rule · time
maxDurationMs > 0 ?
cutoff = current.time − maxDurationMs · idx = history.findIndex(time ≥ cutoff) · keepFrom = min(keepFrom, idx)
rule · tx
toLastTransaction ?
скан назад до последнего события транзакции i · keepFrom = min(keepFrom, i)
end · return
{ steps: history.slice(keepFrom), strategy, position }
currentIndex опускается — текущий снова последний
keepFrom = min(…) — каждое правило может только «отодвинуть» начало окна назад (сделать индекс меньше), но не сузить его. Итог = объединение потребностей всех условий: окна хватает всем сразу, но не больше нужного.

Числовой пример · 10 шагов (индексы 0…9, last = 9)

правилотребованиекандидат idxkeepFrom
countneededSteps = 39 − 2 = 77
timemaxDurationMs → cutoff → idx = 55min(7,5) = 5
txпоследняя транзакция на idx = 88min(5,8) = 5

Итог keepFrom = 5 → history.slice(5) → 5 шагов (5…9). «Прожорливое» time-правило расширило окно назад; более скромное tx-правило проигнорировано через min.

Функция 3 · прогон

processAllStepsAndRecordResults — вся история

Идёт по истории шаг за шагом растущим окном steps[0…i], вызывает processStep на каждом и собирает записи { index, step, result }, дёргая onRecord.

start
processAllStepsAndRecordResults(steps, strategy, { onRecord? })
step
records = []; window = []
для [index, step] ∈ steps
↻ по каждому шагу истории
window.push(step)
окно растёт: steps[0…index]
result = processStep({ steps: window, strategy })
полное окно, без обрезки prepareSteps
record = { index, step, result }
records.push(record) · onRecord?.(record)
end · return
{ records }
Отличие от prepareSteps. Здесь окно намеренно не обрезается — задача собрать результат на каждом шаге по всей доступной истории. prepareSteps же оптимизирует одиночную оценку, отдавая минимальное окно.
Композиция

Как они складываются вместе

Вход и выход prepareSteps — один и тот же тип ProcessStepParams, поэтому функции соединяются в трубу. strategy и position проходят насквозь; position доезжает до ctx и доступен позиционным условиям (TP/SL/PnL).

data
{ steps: history, strategy, position }
prepareSteps
окно из window()
{ steps: обрезанное, strategy, position }
processStep
ctx + AND по CONDITIONS
EvalContext { window, current, position }
result
TradingConditionsStepResult
processStep(prepareSteps({ steps: history, strategy, position }));
Одно место расширения: чтобы добавить условие (например stop_loss), пишешь один ConditionDef с window() и evaluate() и добавляешь в CONDITIONS. processStep, prepareSteps и результат подхватят его автоматически — ядро не меняется.
